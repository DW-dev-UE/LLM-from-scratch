# 생각 실험실 (ThinkingLab)

[← README](../README.md)

---

생각 실험실에 오신 것을 환영합니다.

이곳은 제 이론을 만들고, 그것이 실제로 맞는 이론인지 검증하는 곳입니다.
프로젝트는 GPT-5.6, Claude Fable 5급의 LLM을 개발하는 것입니다.

OpenAI와 Anthropic에 비교하면 저는 지나가는 작은 개미이지만, 다행히도 더 뛰어난 장점이 하나 있습니다.

그건 바보 같은 질문으로 시간을 날려도 되는 것입니다.
일반적인 회사라면 절대 용납되지 않겠지만, 저는 이 바보 같은 질문을 통해 이론을 재정립하는 과정을 매우 좋아합니다.

좋아요, 예를 들어볼게요.

## 가독성이 더 좋은 코딩 AI를 만들려면, 코드를 줄이면 되는 걸까?

초등학교 때부터 코딩을 해온 저는 코드를 어떻게 하면 더 줄일 수 있는가에 대한 연구를 했습니다.
코드가 너무 장황하면, 보기만 해도 읽기 싫어지니까요.

대표적인 생각이 중괄호 제거입니다.

**BEFORE**
```c
if (A >= B)
{
    printf("A >= B");
}
```

**AFTER**
```c
if (A >= B)
    printf("A >= B");
```

간편해졌습니다. 개인적으로는 읽기도 더 쉬워지네요.
여기서 더 줄일 수는 없을까요? 아예 한 줄로 줄여볼게요.

**BEFORE**
```c
if (A >= B)
    printf("A >= B");
```

**AFTER**
```c
if (A >= B) printf("A >= B");
```

뭔가 줄어든 것 같지만, 코드 리뷰는 BEFORE가 더 좋아 보입니다.
어쩌면 중괄호까지 포함된 코드를 더 읽기 쉬워하는 사람이 있을 겁니다.

가독성을 위해 코드를 줄이는 것은 좋은 방법이었지만, 그렇다면 무작정 코드를 줄이는 방법은 잘못되었네요.

어쩌면 개개인의 복잡한 개인만의 규칙은 사용 지침에 넣고, 기본 작업 시에는 모두가 정한 "공식 문서"를 따르는 것이 중요합니다.
그래서 GPT, Claude를 비롯한 여러 LLM은 여전히 중괄호를 넣는 것이겠네요.

이러한 브레인스토밍은 적어도 저 자신에게는 도움이 되는 과정입니다.
그렇다면 이제 더 똑똑한 AI를 만드는 법을 생각해볼게요. 여기에는 철학적인 생각도 들어갑니다.

> [!NOTE]
> 이 폴더에 적힌 결론들은 그 시점의 브레인스토밍 스냅샷입니다. 실제로 채택되었는지, 어떤 스케일부터 적용했는지 같은 "확정된 사실"은 [README](../README.md)를 기준으로 봐 주세요.

## 1. 프로젝트를 진행하는 이유, AGI

인간은 아직 LLM을 몰라요. 어떻게 만드는지만 알고, 왜 지능이 생기는지는 몰라요.

- 왜 그 계산이 지능이 되는가. 가중치들이 어떤 내부 알고리즘을 학습했는지 해석이 불가능합니다.
- 왜 학습이 이렇게 잘 되는가. 경사하강법이 왜 수많은 국소해 중 일반화가 잘 되는 해를 찾는지, 왜 파라미터가 데이터보다 많은데 암기가 아니라 일반화를 하는지, 이론적 설명이 미완성입니다.

심지어는 스케일을 키우면 갑자기 예측하지 못한 능력이 발생하는 경우가 있어요.

> [!TIP]
> Wei et al., *Emergent Abilities of Large Language Models* (2022) 에 따르면, GPT-3·Gopher·Chinchilla 같은 모델들에서 특정 태스크의 성능이 어느 스케일 이전까지는 무작위 수준에 머물다가, 그 지점을 넘으면 갑자기 무작위 이상으로 뛰어오르는 현상이 관찰됩니다.
> 다분야 지식 시험인 MMLU에서는 대략 **70B 파라미터**(Chinchilla 기준) 부근부터 무작위 찍기 수준을 벗어나는 현상이 보고되었습니다.
> ( 단, 이것이 단순히 측정 지표 때문에 생기는 착시인지, 진짜로 능력이 창발하는 것인지는 아직 논쟁거리예요. )

참고로 "예시 몇 개만 보여주면 학습 없이 새 태스크를 수행한다"는 few-shot 능력 자체는 이 논문이 아니라, GPT-3를 처음 소개한 Brown et al., *Language Models are Few-Shot Learners* (2020) 쪽 이야기입니다. 두 논문이 다루는 주장이 다르니 구분해서 기억해두려고요.

LLM 모델에게 기분을 유도하는 방법도 우리는 몰라요.

```text
사람: "오늘 날씨 어때?"
LLM: "오늘 날씨는 화창합니다. '공원 산책'하기 좋은 날이네요."
```

여기서 사용자 메모리에 "사용자는 산책을 좋아하고, 밝은 날씨를 좋아한다"를 넣는 것이 아니라, 정말 날씨가 좋다는 정보만 보고 '공원 산책'을 연관 짓게 할 방법을 우리는 정확히 설명하지 못해요.

**Q. "당신은 몇 살 때부터 의식이 있었나요?"**

많은 사람들이 이 질문을 하면 당황할 겁니다. 그런데 생각해보면 진짜 의식이 생기는 순간은 좀 더 크고 나서예요. 그전에는 기계처럼 행동한다고 생각해요.

발달 관점에서도 맞습니다. 거울 자기인식은 ~18개월부터, 자전적 기억은 ~3세부터 생깁니다. 의식은 스위치가 아니라 점진적으로 켜지는 것처럼 보여요.

애초에 의식이라는 건 뭘까요?
뇌 속에서 수없이 많고 작은 '공'이 이리저리 부딪히면서 '생각'이라는 걸 한다. 그게 '의식'이고 '나'다?

어린아이가 이야기하는 걸 들어보면 재밌는 걸 발견할 수 있어요.

```text
Q. "Alex는 무슨 아이스크림이 좋아요?"
Alex: "Alex는 딸기 아이스크림이 좋아요."
```

어린아이인 Alex는 "나"라는 걸 인지하지 못하고 있어요. 그냥 사람들이 자신을 "Alex"라고 부르니까 그렇게 말하는 것뿐이에요.

저는 개인적으로 이 프로젝트를 진행하면서 이런 가설을 하나 깔고 시작할 거예요.

> "특정 임계점이 넘으면 의식이 생기게 된다. 하지만 그 임계점이 정확히 어디인지, 어떻게 관찰하는지는 아직 모른다."

다만, 어디까지나 가설일 뿐이지 이걸 믿는 건 아니에요. 반론도 정리해둘게요.

- **창발 자체에 대한 논쟁**: 이게 진짜 창발인가, 아니면 측정 지표(metric) 때문에 생기는 착시인가 하는 반론이 있습니다.
- **임계점 = 스케일이라는 전제에 대한 의심**: 단순히 파라미터 수나 FLOPs만 키우는 게 아니라, 아키텍처 변화가 더 중요한 트리거일 가능성이 큽니다.

각 임계점을 어떻게 실험해볼지는 아직 정리된 게 없어서, 이 로그에는 넣지 않았습니다. 실제로 뭔가 시도할 만한 방법이 떠오르면 이 문서를 이어서 채워나갈게요.

## 2. 한정된 자원에서는 다국어를 사용하면 안 된다

H100 한 장으로 7B급 모델을 처음부터 학습하면 (토큰 수·GPU 활용률 가정에 따라 크게 달라지긴 하지만) 대략 6.6년 정도가 걸리는 계산이 나옵니다. 현재의 LLM 모델들은 1T급 이상이기 때문에, 적은 자원으로 효율을 내야 해요.

어쩔 수 없이 모델은 영어권 코퍼스를 위주로 써야 좋을 것 같아요.

이건 사실 새로운 결론은 아니에요. [README](../README.md)의 "326.7M급 모델을 개발하면서 느낀 교훈" 항목에서 이미 "작은 모델 조건부에서 다국어를 지원하는 것은 ~7B급 모델에서는 치명적이다"라고 정리한 적이 있거든요. 여기서는 그 결론이 왜 당연한지를 자원 관점에서 한 번 더 확인해본 것뿐입니다.

## 3. 코딩, 수학 등을 잘하는 LLM을 따로 만들어버리자

굳이 만능 LLM보다 각 분야를 잘하는 LLM을 개발해서 합쳐버리는 게 어떨까 하는 생각을 했어요.
모델 관리 파이프라인을 잘 설계해야겠지만, 시도할 가치가 있어 보여요.

좋아요, 알아봅니다.

이미 있는 아키텍처네요. 별개의 완전한 모델들을 만들어서 inference 때 router나 orchestration layer로 선택·조합하는 방식을 상상했는데, 실제로는 하나의 큰 모델 안에 여러 개의 sub network를 넣는 **MoE (Mixture of Experts)** 형식이었습니다.

각 분야별로 반응하는 FFN을 다중 레이어로 묶는 방식이네요.

```text
MoE Layer ← 여기서 Router가 동작
   ├── Router (Gating Network)
   ├── Expert 1 (FFN)
   ├── Expert 2 (FFN)
   ├── Expert 3 (FFN)
   └── ...
   ↓ (Top-K Expert 출력 합산)
[Output]
```

MoE 아키텍처는 총 파라미터는 엄청나게 많지만 추론 비용은 적고, 각 Expert가 자연스럽게 강해져요.
Mixtral 논문에서도 Expert들이 어느 정도 분화되는 현상이 관찰되었네요.

다만 Router가 제대로 학습되지 않으면 **Expert Collapse** 현상이 발생하니 조심해야 해요.

> [!IMPORTANT]
> Apex-1 초기 학습에는 MoE를 적용하지 않습니다.
> 첫 1B, 새 토크나이저, 새 코퍼스, QK-norm을 채택한 순간부터 변수가 이미 4개나 생겨요. loss가 튀었을 때 원인 후보를 체크할 수 없고, MoE로 전환했을 때 얼마나 좋아졌는지를 재려면 같은 데이터로 학습된 dense 기준점이 먼저 있어야 해요.

## 4. 연산량을 극한으로 최적화하여 더 높은 수준의 모델을 만드는 법

현재 연산은 BF16으로 진행하고 있어요.
다만 이 정도로는 1B 모델을 만드는 데도 너무 오래 걸려요.

모델의 성능을 잃지 않는 선에서, 연산량을 최적화하는 기술을 찾아봐야 해요.

찾아보니 가중치는 절대로 건들면 안 되겠어요. 학습 후반에는 gradient가 매우 작아지는데, FP8로 계산하면 작은 업데이트가 0이 되어버릴 수 있어요.
이건 중대한 문제예요. 오히려 속도를 높이려다가 작업을 잃어버리게 되네요.

하지만 매 스텝마다 계산되는 순전파·역전파에는 FP8을 적용해볼 수 있을 것 같아요.
이건 일시적인 값이고, FP8 학습이 거의 무손실 수준까지 왔다고 해요. 다만 INT8은 아직 이득보다 손해가 압도적으로 크니 배제할게요.

시도할 가치가 충분해요. 이제부터 학습은 FP8로 진행할게요. 다만 여기에도 여러 가지 아키텍처가 있으니 잘 확인하고 진행해야 할 것 같아요.

**Master Weights = BF16 + 연산 = FP8 (Mixed Precision)** 으로 진행합니다.

> [!TIP]
> DeepSeek-V3 기술 보고서(2024)에서도 같은 방향을 확인할 수 있었어요. 행렬곱 연산 대부분은 FP8로 수행하되, embedding · output head · MoE gating · normalization · attention 연산과 master weight·optimizer state는 더 높은 정밀도(BF16/FP32)로 유지하는 방식입니다. 제가 세운 "가중치는 그대로, 연산만 FP8" 원칙과 같은 결론이라 안심이 되네요.
