# 思考実験室（ThinkingLab）

[← README](../README.ja.md)

---

思考実験室へようこそ。

ここは私自身の理論を組み立てて、それが本当に正しい理論なのかを検証する場所です。
プロジェクトの目標は、GPT-5.6、Claude Fable 5 クラスの LLM を開発することです。

OpenAI や Anthropic と比べれば、私は通りすがりの小さなアリのような存在ですが、幸い一つだけ優れている点があります。

それは、バカげた質問に時間を使ってもいいということです。
普通の会社では絶対に許されないことですが、私はこのバカげた質問を通じて理論を組み立て直す過程がとても好きです。

では、例を挙げてみます。

## 読みやすいコーディング AI を作るには、コードを減らせばいいのか？

小学生の頃からコーディングをしてきた私は、コードをどうすればもっと減らせるかを研究してきました。
コードが冗長すぎると、見ただけで読む気が失せますから。

代表的な発想が中括弧の削除です。

**BEFORE**
```c
if (A >= B)
{
    printf("A >= B");
}
```

**AFTER**
```c
if (A >= B)
    printf("A >= B");
```

シンプルになりました。個人的には読みやすくなった気がします。
ここからさらに減らせないでしょうか？ いっそ一行にまとめてみます。

**BEFORE**
```c
if (A >= B)
    printf("A >= B");
```

**AFTER**
```c
if (A >= B) printf("A >= B");
```

何か減ったように見えますが、コードレビューとしては BEFORE の方が良く見えます。
中括弧まで含んだコードの方が読みやすいと感じる人もいるはずです。

可読性のためにコードを減らすこと自体は良い方法でしたが、だからといって闇雲にコードを減らすやり方は間違っていたわけです。

もしかすると、各自の複雑な個人ルールは使用ガイドラインに入れておき、基本作業のときはみんなで決めた「公式ドキュメント」に従うことが重要なのかもしれません。
だから GPT、Claude をはじめとする各種 LLM は今も中括弧を入れるのでしょうね。

こうしたブレインストーミングは、少なくとも私自身にとっては役に立つ過程です。
それでは今度は、もっと賢い AI を作る方法を考えてみます。ここからは哲学的な話も入ってきます。

> このフォルダに書かれている結論は、その時点でのブレインストーミングのスナップショットです。実際に採用されたか、どのスケールから適用したかといった「確定した事実」は、常に [README](../README.ja.md) を基準にしてください。

## 1. このプロジェクトを進める理由、AGI

人間はまだ LLM を理解していません。作り方は分かっていても、なぜ知能が生まれるのかは分かっていません。

- なぜその計算が知能になるのか。重みがどんな内部アルゴリズムを学習したのか、解釈できません。
- なぜ学習がこれほどうまくいくのか。勾配降下法がなぜ無数の局所解の中から汎化性能の高い解を見つけるのか、なぜパラメータ数がデータ数より多いのに暗記ではなく汎化が起きるのか、理論的な説明はまだ未完成です。

さらに、スケールを大きくすると、予測できなかった能力が突然現れることがあります。

> [!TIP]
> Wei et al., *Emergent Abilities of Large Language Models*（2022）によると、GPT-3・Gopher・Chinchilla のようなモデルでは、あるスケールを超えるまでは特定タスクの性能がランダムな水準にとどまり、そのスケールを超えた瞬間に急にランダム以上へ跳ね上がる現象が観察されています。
> 多分野の知識を問う MMLU では、およそ **70B パラメータ**（Chinchilla 基準）付近からランダムな正答率を脱する現象が報告されています。
> （ただし、これが単なる測定指標による錯覚なのか、本当に能力が創発しているのかは、まだ論争の的です。）

なお、「例をいくつか見せるだけで、学習なしに新しいタスクをこなす」という few-shot 能力そのものは、この論文の主張ではなく、GPT-3 を初めて紹介した Brown et al., *Language Models are Few-Shot Learners*（2020）の話です。二つの論文は主張が異なるので、自分の中でも区別しておこうと思います。

LLM に「気分」を誘導する方法も、私たちは分かっていません。

```text
人: 「今日の天気はどう？」
LLM: 「今日は快晴です。『公園の散歩』にちょうどいい日ですね。」
```

ここで大事なのは、ユーザーメモリに「ユーザーは散歩が好きで、晴れた日が好きだ」という情報を入れているわけではなく、単に天気が良いという情報だけから『公園の散歩』を連想させる方法を、私たちは正確には説明できないという点です。

**Q. 「あなたは何歳から意識がありましたか？」**

この質問をされると、多くの人が戸惑うはずです。でも考えてみると、本当に意識が生まれる瞬間は、もっと大きくなってからなんですよね。それ以前は、まるで機械のように振る舞っているのだと思います。

発達心理学の観点から見ても符合します。鏡像自己認知は生後18か月頃から、自伝的記憶は3歳頃からと言われています。意識はスイッチのようなものではなく、徐々に点灯していくもののように見えます。

そもそも「意識」とは何なのでしょうか？
脳の中で無数の小さな「球」があちこちにぶつかり合いながら「思考」というものを作り出す。それが「意識」であり「自分」だ、ということでしょうか？

小さな子どもが話しているのを聞くと、面白いことに気づきます。

```text
Q. 「Alex は何のアイスクリームが好き？」
Alex: 「Alex はいちごアイスクリームが好きです。」
```

幼い Alex はまだ「私」という概念を認識していません。周りの人が自分を「Alex」と呼ぶから、そう話しているだけなのです。

私は個人的に、このプロジェクトを進めるにあたって、次のような仮説を一つ前提として置いておこうと思います。

> 「ある閾値を超えると、意識が生まれる。ただし、その閾値が正確にどこにあるのか、どう観測すればよいのかはまだ分からない。」

ただし、あくまで仮説であって、これを信じているわけではありません。反論も整理しておきます。

- **創発そのものへの論争**: これが本当に創発なのか、それとも測定指標（metric）のせいで生じる錯覚なのか、という反論があります。
- **閾値＝スケールという前提への疑い**: 単純にパラメータ数や FLOPs を増やすだけでなく、アーキテクチャの変化の方がより重要なトリガーである可能性が高いです。

それぞれの閾値をどう実験するかについては、まだ具体案がないのでこのログには入れていません。実際に試せそうな方法を思いついたら、この文書に追記していきます。

## 2. 限られた資源では多言語を使ってはいけない

H100 一枚で 7B クラスのモデルを最初から学習すると、（想定するトークン数や GPU 稼働率によって大きく変わりますが）おおよそ **6.6年** かかる計算になります。現在の LLM モデルは 1T クラス以上が主流なので、限られた資源で効率を出す必要があります。

仕方なく、モデルは英語圏コーパスを中心に据えるのが良さそうです。

これは実は新しい結論ではありません。[README](../README.ja.md) の「326.7M級モデルを開発して得た教訓」の項目で、すでに「小規模モデルという条件下で多言語をサポートするのは、~7B級モデルまでは致命的である」とまとめたことがあります。ここでは、その結論がなぜ当然なのかを、資源という観点からもう一度確認しただけです。

## 3. コーディングや数学が得意な LLM を別々に作ってしまおう

何でもこなす万能 LLM よりも、各分野が得意な LLM を個別に開発して合体させる方がいいのではないか、と考えました。
モデル管理パイプラインをうまく設計する必要はありますが、試す価値はありそうです。

よし、調べてみます。

すでにあるアーキテクチャでした。別々の完全なモデルを作って、推論時に router や orchestration layer で選択・組み合わせる方式を想像していたのですが、実際には一つの大きなモデルの中に複数の sub network を入れる **MoE（Mixture of Experts）** 形式でした。

各分野に反応する FFN を複数レイヤーにわたって束ねる方式ですね。

```text
MoE Layer ← ここで Router が動作
   ├── Router (Gating Network)
   ├── Expert 1 (FFN)
   ├── Expert 2 (FFN)
   ├── Expert 3 (FFN)
   └── ...
   ↓ (Top-K Expert の出力を合算)
[Output]
```

MoE アーキテクチャは総パラメータ数こそ膨大ですが、推論コストは低く、各 Expert が自然と得意分野に強くなっていきます。
Mixtral の論文でも、Expert がある程度分化していく現象が観察されています。

ただし、Router がうまく学習されないと **Expert Collapse** が起きるので注意が必要です。

> [!IMPORTANT]
> Apex-1 の初期学習には MoE を適用しません。
> 最初の 1B、新しいトークナイザー、新しいコーパス、QK-norm を採用した時点で、すでに変数が4つも増えています。loss が跳ねたときに原因の候補を絞り込めなくなりますし、MoE に切り替えてどれだけ良くなったかを測るには、同じデータで学習した dense 基準点が先に必要です。

## 4. 演算量を極限まで最適化して、より高水準のモデルを作る方法

現在の演算は BF16 で進めています。
ただ、この程度では 1B モデルを作るのにも時間がかかりすぎます。

モデルの性能を落とさない範囲で、演算量を最適化する技術を探す必要があります。

調べてみると、重み自体には絶対に手を出してはいけないようです。学習後半では gradient が非常に小さくなりますが、FP8 で計算すると小さな更新量が 0 になってしまうことがあります。
これは重大な問題です。速度を上げようとして、かえって学習の成果を失ってしまいます。

しかし、毎ステップ計算される順伝播・逆伝播には FP8 を適用できそうです。
これは一時的な値であり、FP8 学習はほぼ無損失の水準まで来ているそうです。ただし INT8 はまだ得より損の方が圧倒的に大きいので除外します。

試す価値は十分にあります。これからの学習は FP8 で進めます。ただし、ここにもいくつかのアーキテクチャがあるので、しっかり確認しながら進める必要がありそうです。

**Master Weights = BF16 + 演算 = FP8（Mixed Precision）** で進めます。

> [!TIP]
> DeepSeek-V3 技術レポート（2024）でも同じ方向性が確認できました。行列積演算の大部分は FP8 で行いつつ、embedding・output head・MoE gating・normalization・attention 演算、そして master weight・optimizer state はより高い精度（BF16/FP32）で維持する方式です。「重みはそのまま、演算だけ FP8」という私の原則と同じ結論だったので、少し安心しました。
